好记性不如烂文档

　　曾经一个面试官问我“说说Spring MVC中用到的注解。” 当时不知道是我太紧张，还是太水，就说了个“@Controller”，理直气壮，居然都没有说“@RequestMapping”，哈哈哈…. 书到用时方恨少,我也来说说Spring MVC中的一些比较重要的，至少是应该知道的东西吧，权当是个人使用过程中的总结。
一般开发项目，感觉都是从配置环境开始，没有环境，开发有点难，至少看上去全是bug…
那先说Spring MVC的配置文件，web.xml和spring-mvc.xml

web.xml中一些常用的比较重要的配置

　　在Web容器启动时，加载其他配置文件，作为上下文初始化参数，这个是必要的，因为springMVC几乎都会与其他框架整合，如mybatis、shiro等。当然web容器启动时还会加载listener标签配置的属性。

　　这个配置我认为很重要，至少只要是在国内的项目，能不解决中文编码问题吗？而且这个配置要尽量在web.xml往前放置，否则配置不会生效！
　　至于dispatcherServlet就不赘述了，重要性不言而喻。

Spring-mvc.xml几个比较重要的配置

####Controller中

　　说两个注解@RequestParam和@ModelAttribute
　　(1) @RequsetParam(value=””,required=””)
　　　注解方法参数，以将请求参数绑定到方法参数上，可以指定value的值来匹配请求参数，这时value值和请求参数name应该对应，而name与方法参数可以不一致，required用来设置这个请求参数是否必须的，true为必须，false表示可没有该参数。
如public String login(@RequestParam(value=”username”,required=false)String name,…)则表示前台可以传递name值为username的参数。
　　(2) @ModelAttibute
　　　1) 绑定请求参数到命令对象,有value和required属性放在功能处理方法的入参上时，用于将多个请求参数绑定到一个命令对象，从而简化绑定流程，而且自动暴露为模型数据用于视图页面展示时使用；

只是此处多了一个注解@ModelAttribute(“user”)，它的作用是将该绑定的命令对象以“user”为名称添加到模型对象中供视图页面展示使用。我们此时可以在视图页面使用${user.username}来获取绑定的命令对象的属性。
　　　2)在Spring MVC里，@ModelAttribute通常使用在Controller方法的参数注解中，用于解释model entity，但同时，也可以放在方法注解里。
如果把@ModelAttribute放在方法的注解上时，代表的是：该Controller的所有方法在调用前，先执行此@ModelAttribute方法,若有返回值，则自动将返回值放入model中，传递给url实际请求的方法,并绑定到相应参数。如下代码

Why database migrations?

　　(1)让我们从零开始并假设有一个叫Shiny的项目，它的最初的可交付的产品是一款叫Shiny Soft的软件，它连接的数据是Shiny DB。
　　(2)用以下这个最简洁的图标来代表这个项目
　　　　　　　　　
　　　
　　对于大多数项目而言，它呈现的视图会很快转化为以下这样:

　　
　　现在我们不仅是处理项目环境的一份拷贝，而是一系列这样的拷贝。这就代表大量的挑战。
　　诚然，我们在解决代码方面已经做得很好：
　　　1) 当下版本控制每天都有更好的工具
　　　2) 可再生的构建和连续不断的集成
　　　3) 可以很好地规范发布和部署过程
　　　　
　　　　但是对于数据库而言呢？
　　　　
　　不幸的是到目前为止我们在这方面还没做好。很多项目仍然依赖于人工的应用Sql脚本。而且有的时候还不如这个。这导致许多问题的诞生：
　　1) 在这个机器上的数据库的状态是什么？
　　2) 某个脚本是已经被应用了还是没有被应用？
　　3) 对产品的快速修复已经在后来的测试中被应用？
　　4) 如何新建一个新的数据库实例？
　　对这些问题更多的答案是：我们不知道　　
　　数据库迁移是一个很棒的方式来重新获得对这些麻烦的操作的控制，它允许你：从抓取中重建数据库，在任何时候让数据的状态变得很清晰，以一种明确的方式将当前版本的数据库变更至新的版本

How Flyway works


　　(1) 首先flywaydb会尝试着找出它的元数据表（metadata table），由于数据此时是空的，flyway会改为创建它，默认名为SCHEMA_VERSION，一张简单数据表，它被用来跟踪记录数据库状态。
　　
　　(2)之后flyway会开始扫描应用下的文件系统和类路径以实现迁移（migrations）。可以用sql或java方式实现。对所有的迁移(migrations)以他们的版本号排序和应用。这时，schema_version也根据版本顺序更新，初始化步骤完成。

　　(3)变更数据库到新版本：flyway再次扫描应用下的文件系统和类路径。将这些各个版本的变更与元数据表对照检查，忽略那些版本号小于或等于被标示为当前版本的变更。筛选剩下的变更就是待定的(pending)变更，它们可用，但还未被应用。
　　　　　　　　　
　　(4)之后flyway按照它们的版本号顺序分别执行它们。同时，元数据表schema_version也被更新。
　　

flyway中的约定

图11
　　每次需要升级数据库，无论是数据库结构(DDL)还是更新数据(DML)，只需创建一个新的变更(migration)，满足它的版本号大于当前数据库版本号即可，下一次flyway运行，他将会找到新的变更，并按照此更新数据库。
Flyway 默认规约
　　SQL 脚本文件默认位置是项目的源文件夹下的db/migration 目录。
　　Java 代码默认位于db.migration 包。
　　SQL 脚本文件及Java 代码类名必须遵循以下命名规则：V[][description] 。版本号的数字间以小数点(.)或下划线（_ )分隔开，版本号与描述间以连续的两个下划线（ ）分隔开。如V1_1_0Update.sql 。Java 类名规约不允许存在小数点，所以Java 类名中版本号的数字间只能以下划线进行分隔。

Java的一些重点

发表于 2016-08-29   |   分类于 学习总结   |     |   阅读次数

集合 List、Set、Queue、Map

　　容器是Java提供的持有对象的方式之一，不能持有基本类型，可以指定容器中存放的对象类型(泛型支持),可以自动调整尺寸。
　　List,主要实现有ArrayList,适用于要进行大量的随机访问，底层以数组方式实现；LinkedList，用于经常从表中间插入或删除元素,底层维护Entry节点链表,LinkedList还对Queue提供支持,即实现了Queue接口。
　　Set 不接受重复元素。HashSet提供最快的查询速度，而TreeSet保持元素处于排序状态,LinkedHashSet以插入顺序保存元素。HashSet底层用HashMap实现,Map中的value为一个static、final的Object对象。
　　Map保存相关联的键值对。HashMap用来快速访问;TreeMap保持“键”始终处于排序状态,故没有HashMap快；LinkedHashMap保持元素插入的顺序,但也通过散列提供了快速访问的能力。数组的优势在于遍历，而HashMap等的优势在于快速查找。
　　Vector、HashTable、Stack已过时。

Java的Iterato&迭代器模式

　　迭代器使得，使用容器而不知道或者不关心容器的类型（或结构），成为可能。迭代器是一个对象，它的工作是遍历并选择序列中的对象，客服端程序员不必知道或关心序列底层的结构。
　　Java的Iterator只能单向移动，用来:
　　1)使用方法iterator()要求容器返回一个Iterator。Iterator将准备好返回序列的第一个元素。如果不是为了修改容器本身，使用foreach语法遍历容器更简洁。
　　2)使用next()获得序列中的下一个元素，使用hashNext()检查序列是否还有元素。这可让你不必关心容器中元素的数量，那是由前两个方法关心的了。
　　3)使用remove()将迭代器新近返回的元素删除。调用remove()之前必须先调用next()。
　　容器之间的所有共性都是通过迭代器达成的，实现Collection就意味着需要提供iterator(),但实现Collection就必须实现其所有方法，继承AbstractCollection，花费代价只是稍微减少，因此继承并提供创建迭代器的能力就会显得很重要。

　　生成Itrator是将队列与消费队列的方法连接在一起耦合度最小的方式，与实现Collection相比，在序列类上施加的约束也少得多。
Foreach与迭代器
　　Java SE5引入Iterable接口，该接口包含一个产生Iterator的iterator()方法，并且Iterable接口被foreach用来在序列中移动，若创建实现Iterable的类，都可以将它用于foreach语句中。
　　在Java SE5中大量的类都是Iterable类型，主要包括所有的Collection类（但不包括各种Map）,foreach可用于数组或其他任何Iterable，但并不意味着数组肯定也是一个Iterable。

equals()&hashCode()

Obeject的equals()方法

　　超类Object中有这个equals()方法，故所有类继承这个方法，该方法主要用于比较两个对象是否相等。所有的对象都拥有标识(内存地址)和状态(数据)，同时“==”比较两个对象的的内存地址，所以说使用Object的equals()方法是比较两个对象的内存地址是否相等，即若object1.equals(object2)为true，则表示equals1和equals2实际上是引用同一个对象。

String的equals方法

在Java规范中，它对equals()方法的使用必须要遵循如下几个规则：
　　equals 方法在非空对象引用上实现相等关系：
　　1、自反性：对于任何非空引用值 x，x.equals(x) 都应返回 true。
　　2、对称性：对于任何非空引用值 x 和 y，当且仅当 y.equals(x) 返回 true 时，x.equals(y) 才应返回 true。
　　3、传递性：对于任何非空引用值 x、y 和 z，如果 x.equals(y) 返回 true，并且 y.equals(z) 返回 true，那么 x.equals(z) 应返回 true。
　　4、一致性：对于任何非空引用值 x 和 y，多次调用 x.equals(y) 始终返回 true 或始终返回 false，前提是对象上 equals 比较中所用的信息没有被修改。
　　5、 对于任何非空引用值 x，x.equals(null) 都应返回 false。
　　对于上面几个规则，我们在使用的过程中最好遵守，否则会出现意想不到的错误。
　　在覆写equals()方法时，一般都是推荐使用getClass来进行类型判断，不是使用instanceof。instanceof判断其左边对象是否为其右边类的实例，返回boolean类型的数据，也可以用来判断继承中的子类的实例是否为父类的实现。
　　Obeject的hashCode()方法
　　hashCode()方法将该对象的内部地址转换成一个整数(哈希码)，在调用时返回，方法定义为public native int hashCode();
Equals 方法与HashCode联系
　　Object的hashCode 的常规协定是：
　　在 Java 应用程序执行期间，在对同一对象多次调用 hashCode 方法时，必须一致地返回相同的整数，前提是将对象进行 equals比较时所用的信息没有被修改。从某一应用程序的一次执行到同一应用程序的另一次执行，该整数无需保持一致。
　　如果根据 equals(Object) 方法，两个对象是相等的，那么对这两个对象中的每个对象调用 hashCode 方法都必须生成相同的整数结果。
　　如果根据 equals(java.lang.Object)方法，两个对象不相等，那么对这两个对象中的任一对象上调用hashCode方法不要求一定生成不同的整数结果。但是，程序员应该意识到，为不相等的对象生成不同整数结果可以提高哈希表的性能。
　　在Map中散列使得查询得以快速的进行。散列将键保存在某处，以便能很快找到。存储一组元素最快的数据结构是数组，所以用它来表示键的信息（键的信息，而不是键本身）。
　　数组并不保存键本身，而是通过键对象生成一个数字，将其作为数组的下标，这个数字就是散列码，由Object或你自己的类覆盖的hashCode()方法生成。

需要注意记住的事情

　　尽量保证使用对象的同一个属性来生成hashCode()和equals()两个方法。在我们的案例中,我们使用员工id。
　　eqauls方法必须保证一致（如果对象没有被修改，equals应该返回相同值）
　　任何时候只要a.equals(b),那么a.hashCode()必须和b.hashCode()相等。
两者必须同时重写。
　　当使用ORM的时候特别要注意的如果你使用ORM处理一些对象的话，你要确保在hashCode()和equals()对象中使用getter和setter而不是直接引用成员变量。因为在ORM中有的时候成员变量会被延时加载，这些变量只有当getter方法被调用的时候才真正可用。例如在我们的例子中，如果我们使用e1.id==e2.id则可能会出现这个问题，但是我们使用e1.getId()==e2.getId()就不会出现这个问题。
