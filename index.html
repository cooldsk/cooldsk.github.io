Shiro——登录，岂止用户名和密码

发表于 2016-08-31   |   分类于 学习总结   |     |   阅读次数

　　当然，所有这些技术，网上都有很多详细的教程，五花八门，中英文，可能需要自己筛选，在此我仅作为一个初学者，分享一下学习重点和收获。

首先，shiro是什么？

　　　　
　　Apache Shiro（发音为“shee-roh”，日语“堡垒（Castle）”的意思）是一个强大易用的Java安全框架，提供了认证、授权、加密和会话管理功能，可为任何应用提供安全保障 - 从命令行应用、移动应用到大型网络及企业应用。

其次，shiro能做什么？

　　
　　Shiro为解决下列问题（它们常被称为应用安全的四要素）提供了保护应用的API：
　　• 认证 - 用户身份识别，常被称为用户“登录”；
　　• 授权 - 访问控制；
　　• 密码加密 - 保护或隐藏数据防止被偷窥；
　　• 会话管理 - 每用户相关的时间敏感的状态。
　　• Shiro还支持一些辅助特性，如Web应用安全、单元测试和多线程，它们的存在强化了上面提到的四个要素。
　　shiro的优势就不赘述了，这种东西不太好说，什么易用性、广泛性、可插拨之类的。

我们从外部来看，Shiro如何完成工作？

　　　　　
　　　　　　　核心概念：Subject，SecurityManager和Realms
Subject
　　在考虑应用安全时，你最常问的问题可能是“当前用户是谁？”或“当前用户允许做X吗？”。当我们写代码或设计用户界面时，问自己这些问题很平常：应用通常都是基于用户故事构建的，并且你希望功能描述（和安全）是基于每个用户的。所以，对于我们而言，考虑应用安全的最自然方式就是基于当前用户。Shiro的API用它的Subject概念从根本上体现了这种思考方式。
Subject一词是一个安全术语，其基本意思是“当前的操作用户”。称之为“用户”并不准确，因为“用户”一词通常跟人相关。在安全领域，术语“Subject”可以是人，也可以是第三方进程、后台帐户（Daemon Account）或其他类似事物。它仅仅意味着“当前跟软件交互的东西”。但考虑到大多数目的和用途，你可以把它认为是Shiro的“用户”概念。在代码的任何地方，你都能轻易的获得Shiro Subject，参见如下代码：

SecurityManager
　　Subject的“幕后”推手是SecurityManager。Subject代表了当前用户的安全操作，SecurityManager则管理所有用户的安全操作。它是Shiro框架的核心，充当“保护伞”，引用了多个内部嵌套安全组件，它们形成了对象图。但是，一旦SecurityManager及其内部对象图配置好，它就会退居幕后，应用开发人员几乎把他们的所有时间都花在Subject API调用上。
那么，如何设置SecurityManager呢？嗯，这要看应用的环境。例如，Web应用通常会在Web.xml中指定一个Shiro Servlet Filter，这会创建SecurityManager实例，如果你运行的是一个独立应用，你需要用其他配置方式，但有很多配置选项。
　　一个应用几乎总是只有一个SecurityManager实例。它实际是应用的Singleton（尽管不必是一个静态Singleton）。跟Shiro里的几乎所有组件一样，SecurityManager的缺省实现是POJO，而且可用POJO兼容的任何配置机制进行配置 - 普通的Java代码、Spring XML、YAML、.properties和.ini文件等。基本来讲，能够实例化类和调用JavaBean兼容方法的任何配置形式都可使用。
为此，Shiro借助基于文本的INI配置提供了一个缺省的“公共”解决方案。INI易于阅读、使用简单并且需要极少依赖。你还能看到，只要简单地理解对象导航，INI可被有效地用于配置像SecurityManager那样简单的对象图。
Realms
　　Shiro的第三个也是最后一个概念是Realm。Realm充当了Shiro与应用安全数据间的“桥梁”或者“连接器”。也就是说，当切实与像用户帐户这类安全相关数据进行交互，执行认证（登录）和授权（访问控制）时，Shiro会从应用配置的Realm中查找很多内容。
从这个意义上讲，Realm实质上是一个安全相关的DAO：它封装了数据源的连接细节，并在需要时将相关数据提供给Shiro。当配置Shiro时，你必须至少指定一个Realm，用于认证和（或）授权。配置多个Realm是可以的，但是至少需要一个。
　　Shiro内置了可以连接大量安全数据源（又名目录）的Realm，如LDAP、关系数据库（JDBC）、类似INI的文本配置资源以及属性文件等。如果缺省的Realm不能满足需求，你还可以插入代表自定义数据源的自己的Realm实现。

从 Shiro 内部来看下 Shiro 的架构

　　　　
　　Subject：主体，可以看到主体可以是任何可以与应用交互的 “用户”；
　　SecurityManager：相当于 SpringMVC 中的 DispatcherServlet 或者 Struts2 中的 FilterDispatcher；是 Shiro 的心脏；所有具体的交互都通过
SecurityManager 进行控制；它管理着所有Subject、且负责进行认证和授权、及会话、缓存的管理。
　　Authenticator：认证器，负责主体认证的，这是一个扩展点，如果用户觉得 Shiro 默认的不好，可以自定义实现；其需要认证策略（Authentication Strategy），即什么情况下算用户认证通过了；
　　Authrizer：授权器，或者访问控制器，用来决定主体是否有权限进行相应的操作；即控制着用户能访问应用中的哪些功能；
　　Realm：可以有 1 个或多个 Realm，可以认为是安全实体数据源，即用于获取安全实体的；可以是 JDBC 实现，也可以是 LDAP 实现，或者内存实现等等；由用户提供；注意：Shiro 不知道你的用户 / 权限存储在哪及以何种格式存储；所以我们一般在应用中都需要实现自己的 Realm；
　　SessionManager：如果写过 Servlet 就应该知道 Session 的概念，Session 呢需要有人去管理它的生命周期，这个组件就是 SessionManager；而 Shiro 并不仅仅可以用在 Web 环境，也可以用在如普通的 JavaSE 环境、EJB 等环境；所有呢，Shiro 就抽象了一个自己的 Session 来管理主体与应用之间交互的数据；这样的话，比如我们在 Web 环境用，刚开始是一台 Web 服务器；接着又上了台 EJB 服务器；这时想把两台服务器的会话数据放到一个地方，这个时候就可以实现自己的分布式会话（如把数据放到 Memcached 服务器）；
　　SessionDAO：DAO 大家都用过，数据访问对象，用于会话的 CRUD，比如我们想把 Session 保存到数据库，那么可以实现自己的 SessionDAO，通过如 JDBC 写到数据库；比如想把 Session 放到 Memcached 中，可以实现自己的 Memcached SessionDAO；另外 SessionDAO 中可以使用 Cache 进行缓存，以提高性能；
　　CacheManager：缓存控制器，来管理如用户、角色、权限等的缓存的；因为这些数据基本上很少去改变，放到缓存中后可以提高访问的性能
　　Cryptography：密码模块，Shiro 提高了一些常见的加密组件用于如密码加密 / 解密的。
　　接触shiro，了解以上内容是必要的。

　　1) 在这个机器上的数据库的状态是什么？
　　2) 某个脚本是已经被应用了还是没有被应用？

　　4) 如何新建一个新的数据库实例？
　　对这些问题更多的答案是：我们不知道　　
　　数据库迁移是一个很棒的方式来重新获得对这些麻烦的操作的控制，它允许你：从抓取中重建数据库，在任何时候让数据的状态变得很清晰，以一种明确的方式将当前版本的数据库变更至新的版本

How Flyway works


　　(1) 首先flywaydb会尝试着找出它的元数据表（metadata table），由于数据此时是空的，flyway会改为创建它，默认名为SCHEMA_VERSION，一张简单数据表，它被用来跟踪记录数据库状态。
　　
　　(2)之后flyway会开始扫描应用下的文件系统和类路径以实现迁移（migrations）。可以用sql或java方式实现。对所有的迁移(migrations)以他们的版本号排序和应用。这时，schema_version也根据版本顺序更新，初始化步骤完成。

　　(3)变更数据库到新版本：flyway再次扫描应用下的文件系统和类路径。将这些各个版本的变更与元数据表对照检查，忽略那些版本号小于或等于被标示为当前版本的变更。筛选剩下的变更就是待定的(pending)变更，它们可用，但还未被应用。
　　　　　　　　　
　　(4)之后flyway按照它们的版本号顺序分别执行它们。同时，元数据表schema_version也被更新。
　　

flyway中的约定

图11
　　每次需要升级数据库，无论是数据库结构(DDL)还是更新数据(DML)，只需创建一个新的变更(migration)，满足它的版本号大于当前数据库版本号即可，下一次flyway运行，他将会找到新的变更，并按照此更新数据库。
Flyway 默认规约
　　SQL 脚本文件默认位置是项目的源文件夹下的db/migration 目录。
　　Java 代码默认位于db.migration 包。
　　SQL 脚本文件及Java 代码类名必须遵循以下命名规则：V[][description] 。版本号的数字间以小数点(.)或下划线（_ )分隔开，版本号与描述间以连续的两个下划线（ ）分隔开。如V1_1_0Update.sql 。Java 类名规约不允许存在小数点，所以Java 类名中版本号的数字间只能以下划线进行分隔。

我理解的设计模式，好像有点缺乏实践

发表于 2016-06-20   |   分类于 学习总结   |     |   阅读次数

单例模式

　　Ensure a class has only instance, and provide a global point of access to it.
　　确保一个类只有一个实例，而且自行实例化并向整个系统提供这个实例。
通过私有的构造函数确保在一个应用中只产生一个实例，并且是自行实例化的。

饿汉式单例（线程安全）

　　这种方式没有使用同步，并且确保了调用static getInstance()方法时才创建Singleton的引用（static 的成员变量在一个类中只有一份）。饿汉式的实现方式虽然貌似开销比较大，但是不会出现线程安全的问题，也是解决线程安全的单例实现的有效方式。

懒汉式单例（线程安全）

　　但很多时候我们通常会认为锁定整个方法的是比较耗费资源的，称为重量级锁，代码中实际会产生多线程访问问题的只有 instance = new Singleton．

二次检查的单例

　　“双重检查锁定背后的理论是完美的。不幸地是，现实完全不同。双重检查锁定的问题是：并不能保证它会在单处理器或多处理器计算机上顺利运行。双重检查锁定失败的问题并不归咎于 JVM 中的实现 bug，而是归咎于 Java 平台内存模型。内存模型允许所谓的“无序写入”，这也是这些习语失败的一个主要原因。”
　　优点
　　对于一个对象需要频繁地创建、销毁，而无法优化这些操作的性能的时候，使用单例模式优势明显。
　　减少系统的性能开销，产生一个对象需要比较多的资源和开销时，在应用启动时直接产生一个单例对象，然后用永久驻留内存的方式来解决。
　　避免对资源的多重使用，也可在系统设置全局的访问点，例如设计一个单例类负责所有数据表的映射处理。
　　Spring中Bean默认就是单例的，这样使得spring容器可以管理这些Bean的生命周期。

工场方法模式

　　工厂方法模式使用频率非常高，其定义为:
　　Define an interface for creating an object, but let subclasses decide which class to instaniate. Factory Method lets a class defer instantiation to subclasses.定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。
　　具有良好的封装性，代码结构清晰。扩展性好，在增加产品类的情况下，只需修改具体工厂类或扩展一个工厂类即可。
屏蔽产品类。产品类如何变化，调用者并不关心，它只是依赖于产品的接口，只要接口不变，系统中的上层模块就不要变化。
工厂方法模式是new一个对象的替代品，应慎重考虑是否要增加一个工厂类进行管理，增加代码的负责度。

抽象工厂模式

　　Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
　　为创建一组相关或相互依赖的对象提供一个接口，而无须指定它们的具体类。抽象工厂模式是工厂方法模式的升级版本，在有多个业务品种、业务分类时，通过抽象工厂模式产生需要的对象是一种非常好的解决方式。
　　使用场景：一个对象族（或是一组没有任何关系的对象）都有相同的约束。
　　如一个应用，需要在三个不同平台运行，通过抽象工厂模式屏蔽掉操作系统对应用的影响，三个不同操作系统上的软件功能、应用逻辑、UI都应该非常类似，唯一不同的是调用不同的工厂方法，由不同的产品类去处理与操作系统交互的信息。

模板方法模式

　　Define the skeleton of an algorithm in an operation, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm’s structure.
　　定义一个操作中的算法的框架，而将一些步骤延迟到子类中。使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。
　　我们知道一个算法所需的关键步骤，并确定了这些步骤的执行顺序。但是某些步骤的具体实现是未知的，或者说某些步骤的实现与具体的环境相关。
　　例子1：银行业务办理流程。在银行办理业务时，一般都包含几个基本固定步骤:
　　取号排队->办理具体业务->对银行工作人员进行评分。
　　取号取号排队和对银行工作人员进行评分业务逻辑是一样的。但是办理具体业务是个不相同的，具体业务可能取款、存款或者转账。
　　
　　抽象类（AbstractClass）: 定义抽象的原语操作（primitive operation） ，具体的子类将重定义它们以实现一个算法， 实现一个模板方法,定义一个算法的骨架。该模板方法不仅调用原语操作，也调用定义
　　具体子类 （ConcreteClass）:实现原语操作以完成算法中与特定子类相关的步骤。
　　模板方法模式封装不变的部分，扩展可变部分。
　　提取公共部分代码，便于维护。
　　行为由父类控制，子类实现。
　　模板方法模式与控制反转（好莱坞原则）在模板方法模式中，子类不显式调用父类的方法，而是通过覆盖父类的方法来实现某些具体的业务逻辑，父类控制对子类的调用，这种机制被称为好莱坞原则(Hollywood Principle)，好莱坞原则的定义为：“不要给我们打电话，我们会给你打电话(Don’t call us, we’ll call you)”。在好莱坞，把简历递交给演艺公司后就只有回家等待。由演艺公司对整个娱乐项的完全控制，演员只能被动式的接受公司的差使,在需要的环节中，完成自己的演出。模板方法模式充分的体现了“好莱坞”原则。由父类完全控制着子类的逻辑，子类不需要调用父类，而通过父类来调用子类，子类可以实现父类的可变部份，却继承父类的逻辑,不能改变业务逻辑。

策略模式(Strategy Pattern)

　　比较简单的模式，也叫政策模式，但是需要实践。Define a family of algorithms,encapsulate each one, and make them interchangeable.定义一组算法，将每个算法都封装起来，并且是它们之间可以互换。

操作系统中的一些概念

发表于 2016-06-18   |   分类于 学习总结   |     |   阅读次数

堆和栈

　　由编译器自动分配和释放，存放函数的参数值、局部变量的值，操作方式类似于数据结构中的栈（后进栈，先出栈）
　　一般由程序员分配和释放，若程序员不释放，程序结束时可能由OS回收，与数据结构中的堆是两码事，操作方式类似于链表

进程与线程

　　1)通俗的讲进程可以看做是执行中的程序，但它包含程序代码、程序计数器和寄存器的值（即当前活动）、堆栈段（stack,包括临时数据，如函数参数，局部参数和返回地址 ）、数据段（包括全局变量），还可能包含堆(heap,是在进程运行期间动态分配的内存）。
进程是活动的实体，当可执行文件被装入内存时，一个程序才能成为进程。
　　2)进程是大多数系统中的工作单元，系统由一组进程组成：操作系统进程执行系统代码，用户进程执行用户代码，所有这些进程可以并发执行，通过进程之间切换，操作系统能使计算机运行更高效。进程管理也由操作系统负责，目前大多数现代操作系统支持多线程进程。
　　3)进程有新建、就绪、运行、等待、终止等状态，一次只有一个进程在一个处理上运行，但多个进程可处于就绪或等待状态。

进程状态

新建：进程正在被创建
　　就绪：进程等待分配CPU资源和时间片
　　运行：指令正在被执行，最终完成，或被中断或等待特定事件发生（I/O请求）
　　等待：进程等待某个事件的发生（如I/O完成或收到信号）
　　终止：进程完成执行
　　每个进程在操作系统内用进程控制块（process control block,PCB,也称为任务控制块）表示。

进程调度

　　进程被放到大容量存储设备（通常为磁盘）的缓冲池中，保存在那里以便以后执行。长期调度程序（或作业调度程序）从该池中选择进程，并装入内存已准备执行，必须合理选择一组包含以I/O为主的和以CPU为主的进程组合，通常数分钟执行一次，取决于多道程序设计的程度，有些系统可能没有或很少有长期调度程序，如UNIX和微软Windows的分时系统。短期调度程序（CPU调度程序）从准备执行的进程中选择进程，并为之分配CPU，通常没100ms至少执行一次，故要求短期调度程度必须要快。
　　将CPU切换到另一个进程需要保存当前进程的状态并恢复另一个进程的状态，这一任务称为上下文切换。

进程创建

　　绝大多数系统内的进程能并发执行，它们可以动态创建和删除。通过创建进程系统调用创建多个新进程，创建进程称为父进程，而新进程称为子进程。
进程通唯一的进程标识付（process identifier, pid）来标识，pid通常是一个整数值。
　　进程需要一定的资源来完成其任务，在一个进程创建子进程时：
　　　1）子进程可能从操作系统那里直接获得资源
　　　2）可能从父进程获得资源。
　　　3）父进程可能必须在其子进程之间分配资源或共享资源（如内存或文件），限制子进程只能使用父进程的资源能防止创建过多的进程带来的系统超载。
　　　4）初始化数据（或输入）由父进程传递给子进程。
　　进程创建新进程时，有两种执行可能：
　　　1）父进程与子进程并发执行
　　　2）父进程等待，直到某个或全部子进程执行完。
　　新进程地址空间有两种可能：
　　　1）子进程是父进程的复制品，即具有与父进程相同的程序和数据。
　　　2）子进程装入另一个新程序。
　　UNIX操作系统中，通过fork()系统调用，可创建新进程。新进程通过复制原来进程的地址空间而成。这允许父子进程方便地通信。两个进程都继续执行位于系统调用fork()之后的指令。区别是对子进程，系统调用fork()的返回值为0；对父进程，返回值为子进程的进程标识符（非零）。
　　在系统调用fork()之后，一个进程会使用系统调用exex()，以用新程序来取代进程的地址空间。Exec()将二进制文件装入内存，并开始执行。父进程能创建更多的子进程，或者在子进程运行时没有什么可做，调用wait()把自己移出就绪队列等待子进程终止。
　　Win32 API采用CreateProcess()（与fork()类似）函数创建进程，需要将一个特殊程序装入子进程的地址空间，与fork()不需要传递参数不同，CreateProcess()至少要10个参数。

进程终止

　　当进程执行完成最后的语句并使用系统调用exit()请求操作系统删除自身时，进程终止。此时进程可以返回状态值（常为整数）到父进程（通过系统调用wait()）。所有进程资源都会被操作系统释放。
　　父进程可以通过适当的系统调用（如win32的TerminateProcess()）来终止其子进程，父进程需要知道所有子进程的标识符，因此在创建子进程时，子进程标识符要传给父进程。
　　当子进程使用了超过它分配到的一些资源，或分配给它的任务不再需要，或者父进程退出时，父进程会终止其子进程。如果一个进程终止（正常或不正常），那么它所有子进程也将终止，这称为级联终止（如VMS系统）。UNIX系统中系统调用wait()返回了终止子进程的标识符，以使父进程知道哪个子进程终止了，如果父进程终止，那么所有子进程以init进程作为父进程，因此子进程仍有一个父进程收集状态和执行统计。

进程间通信

　　如果系统中一个进程能影响其他进程或被其他进程所影响，那么该进程是协作的，反正为独立的，协作进程需要一种进程间通信机制（interprocess communication,IPC）来允许进程相互交换数据与信息。
进程间通信有两种基本模式: 1)共享内存 2)消息传递
　　1)共享内存
　　 建立一块供协作进程共享的内存区域，进程间通过向此共享区域读或写入数据来交换信息。共享内存比消息传递快，消息传递系统通常需要系统调用来实现，因此需要更多的内核介入的时间消耗。在共享内存系统中，仅在建立共享内存区域时需要系统调用。
　　2)消息传递
　　通过在协作进程间交换信息来实现通信。消息传递对于交换较少数量的数据很有用，因为不需要避免冲突。对于计算机的通信，消息传递也比共享内存更容易实现。
　　共享内存系统
　　采用共享内存的进程间通信需要通信进程建立共享内存区域。通常一块共享内存区域驻留在生成共享内存段进程的地址空间。其他希望使用此共享内存段进行通信的进程必须将此放到它们自己的地址空间。共享内存需要取消 操作系统试图阻止一个进程范围另一个进程的内存 这一限制，进程间通过在共享内存区域内读或写来交换信息，并保证它们不会向同一区域同时写数据。
　　内存共享是解决生产者-消费者问题方法中的一种。使用一个缓冲（驻留在生产者进程和消费者进程的共享内存区域内）来被生产者填充并被消费者使用。
　　消息传递系统
　　由操作系统提供机制，让协作进程能通过消息传递工具来进行通信。在分布式环境中特别有用。如果进程P和Q需要通信，那么它们必须彼此相互发送消息和接收消息，它们之间必须要有通信线路。
　　逻辑实现线路和send()/receive()操作的方法:
　　　1)直接或间接通信
　　　2)同步或异步通信

TCP与UDP区别？

　　(1) TCP(Transmission Control Protocol 传输控制协议)，是面向连接的，提供可靠数据传输服务，而UDP(User Datagram Protocol，用户数据报协议)，是无连接的，两个进程通信前没有握手过程，提供的是不可靠数据传输服务，会出现丢包。
　　(2) TCP协议还具有拥塞控制机制，抑制发送方进程，限制每个TCP连接，使它们达到公平共享网络带宽。而UDP没有拥塞控制机制，发送端可以以任何速率向网络层注入数据，适合用于有最低带宽限制的实时音频/视频通信。
　　(3) UDP分组首部开销比TCP小，每个TCP首部有20字节开销，而UDP只有8字节。

TCP如何保证可靠传输？三次握手过程？

　　(1) TCP报文段的首部提供序列号，以便接收方根据序列号是否相同判定是否出现数据冗余
　　(2)TCP只提供肯定确认（ACK），通过TCP报文段首部确认号体现，即发送方期望从接收方收到的下一个字节的序号
　　(3)TCP报文段提供首部校验和，检测一个传输分组是否出现比特错误。
　　(4)当发送方向接收方传输分组时。出现丢包或者比特差错后，接受方都不会返回发送方发送这个分组时期待的确认号。因·此在TCP中发送方为每个分组提供单一计时器，当发送方发送分组时总是会启动对应的计时器，在计时器超时后发送方如果还没有收到期待的确认号，就会重传该分组，并且采用加倍超时间隔机制，每次重传都会把计时器下一次超时间隔加倍。TCP也采用快速重传的机制，即接受方如果收到比期望序号大的报文段时，就会立即发送三个冗余ACK，这个ACK就是期待接受的ACK号（比刚收到的序号小），当发送端接收到三个冗余ACK后，即使计时器未超时，也会重传对应序号报文段。
　　(5)TCP采用累积确认方式，发送方如果收到更大ACK后会认为比该ACK小的报文段都收到了，即接收方返回的更小ACK丢失情况下。但接收方收到无序分组时而且中间某一个分组丢失时，仅要求重传该分组，即选择确认。TCP差错恢复机制也行被划分为回退N协议和选择重传的混合体。
三次握手过程？
　
　　①客户端首先向服务器端发送一个特殊TCP报文，该报文不包含应用层数据，也称为SYN报文段，报文首部，SYS标志位被置1，序号字段被一个起始序列号（client-isn）填充，最后将该报文段封装在IP数据包中发送给服务器。
　　②服务器端接受到该报文段后，为TCP连接分配缓存和变量，并向客户端发送允许连接报文段，也被成为SYNACK报文段，该报文段SYN比特位被置1，首部确认号为接收到的报文段序号+1，服务器选择自己初始序号并放置在首部序号字段。
　　③客服端收到SYNACK报文段后也要为该连接分配缓存和变量，并回复报文段以对允许连接的报文进行确认，由于连接已经建立，故SYN位被置为0，确认号为收到的SYNACK报文序号+1.三次握手完成
四次分手过程？
　　　　　　　　　
　　①参与连接的两个进程中的任何一个都能终止连接
　　②若客户端发起关闭，则发送一个特殊TCP报文段,该报文段首部FIN标志位被置1，进入FIN_WAIT_1状态，此状态等待服务器回复带有确认信息的报文段，服务器收到该报文段后向客户端回送一个确认报文段，客户端进入FIN_WAIT_2状态，等待服务器发送FIN位被置1的报文段。
　　③之后，服务器发送终止报文段，其FIN比特位被置1，客户端对这个服务器的终止报文段进行确认，进入TIME_WAIT状态，使得客户端能够重传确认报文，防止ACK丢失，时间一般为30秒，1或2分钟。

HashMap的实现

　　不同的键可以产生相同的下标（哈希码），即可能会有冲突，可由外部链接处理，数组并不保存至，而是保存值得List（或者链表），散列函数较好的话，每个位置就只有较少的值，故可以快速跳到数组的某个位置，只对很少的元素用equals()方法进行比较（保存在List中元素）。这便是HashMap如此快的原因。
　　HashMap是基于哈希表的Map接口的非同步实现。此实现提供所有可选的映射操作，并允许使用null值和null键。此类不保证映射的顺序，特别是它不保证该顺序恒久不变。
　　在java编程语言中，最基本的结构就是两种，一个是数组，另外一个是模拟指针（引用），所有的数据结构都可以用这两个基本结构来构造的，HashMap也不例外。HashMap实际上是一个“链表散列”的数据结构，即数组和链表的结合体。
　　　　　　　　　　　　HashMap底层实现原理
　　hash(int h)方法根据key的hashCode重新计算一次散列。此算法加入了高位计算，防止低位不变，高位变化时，造成的hash冲突。于任意给定的对象，只要它的 hashCode() 返回值相同，那么程序调用 hash(int h) 方法所计算得到的 hash 码值总是相同的。
　　当我们往HashMap中put元素的时候，先根据key的hashCode重新计算hash值，根据hash值得到这个元素在数组中的位置（即下标 通过hash&length-1计算），如果数组该位置上已经存放有其他元素了，那么在这个位置上的元素将以链表的形式存放，新加入的放在链头，最先加入的放在链尾（java SE5，jdk1.8将新加入的Node(Entry类型)放在链尾）。如果数组该位置上没有元素，就直接将该元素放到此数组中的该位置上。
　　HashMap 在底层将 key-value 当成一个整体进行处理，这个整体就是一个 Entry 对象。HashMap 底层采用一个 Entry[] 数组来保存所有的 key-value 对，当需要存储一个 Entry 对象时，会根据hash算法来决定其在数组中的存储位置，在根据equals方法决定其在该数组位置上的链表中的存储位置；当需要取出一个Entry时，也会根据hash算法找到其在数组中的存储位置，再根据equals方法从该位置上的链表中取出该Entry。

HashMap和Hashtable的区别

　　HashMap和Hashtable都实现了Map接口，主要的区别有：线程安全性，同步(synchronization)，以及速度。
　　　1)HashMap可以接受为null的键值(key)和值(value)，而Hashtable则不行。HashMap底层有一个Node(实现了Map.Entry)内部类，key和value封装在Node中。
　　 2)HashMap是非synchronized，而Hashtable是synchronized，这意味着Hashtable是线程安全的，多个线程可以共享一个Hashtable；Java5提供了ConcurrentHashMap，它是HashTable的替代，比HashTable的扩展性更好。
　　 3) 另一个区别是HashMap的迭代器(Iterator)是fail-fast(快速报错)迭代器，而Hashtable的enumerator迭代器不是fail-fast的。所以当有其它线程改变了HashMap的结构（增加或者移除元素），将会抛出ConcurrentModificationException，但迭代器本身的remove()方法移除元素则不会抛出ConcurrentModificationException异常。但这并不是一个一定发生的行为，要看JVM。这条同样也是Enumeration和Iterator的区别。
　　4）由于Hashtable是线程安全的也是synchronized，所以在单线程环境下它比HashMap要慢。如果你不需要同步，只需要单一线程，那么使用HashMap性能要好过Hashtable。
　　5）HashMap不能保证随着时间的推移Map中的元素次序是不变的。
　　6) 哈希值的使用不同，HashTable直接使用对象的hashCode, HashMap重新计算hash值，加入了高位计算，而且用与(hash&length-1)代替求模得到在数组中索引index。&比%具有更高的效率。

　　要注意的一些重要术语：
　　1) sychronized意味着在一次仅有一个线程能够更改Hashtable。就是说任何线程要更新Hashtable时要首先获得同步锁，其它线程要等到同步锁被释放之后才能再次获得同步锁更新Hashtable。
　　2) Fail-fast和iterator迭代器相关。如果某个集合对象创建了Iterator或者ListIterator，然后其它的线程试图“结构上”更改集合对象，将会抛出ConcurrentModificationException异常。但其它线程可以通过set()方法更改集合对象是允许的，因为这并没有从“结构上”更改集合。但是假如已经从结构上进行了更改，再调用set()方法，将会抛出IllegalArgumentException异常。
　　3) 结构上的更改指的是删除或者插入一个元素，这样会影响到map的结构。
　　我们能否让HashMap同步？
　　Collections类有办法能够自动同步整个容器

　　结论
　　Hashtable和HashMap有几个主要的不同：线程安全以及速度。仅在需要完全的线程安全的时候使用Hashtable，而如果你使用Java 5或以上的话，请使用ConcurrentHashMap吧。

Java泛型(Generics)概述

　　(1)使代码能够应用于“某种不具体的类型”,而不局限于一个具体的类或接口。
　　(2)泛型实现了参数化类型的概念，使代码可以应用于多种类型。
　　(3)当创建参数化类型的一个实例时，编译器会进行转型操作，并且保证类型的正确性。
　　(4)使用别人创建的泛型容易，自己创建泛型会吃惊，因为在Java中它并不那么强大。
　　(5)Java泛型灵感来自C++，有局限，但可以使得代码更优雅
　　(6)核心概念:告诉编译器想使用什么类型，然后编译器帮助你处理一切细节。
　　泛型方法

型参数声明部分在方法返回类型之前（在下面例子中的）。

　　　(3)类型参数能被用来声明返回值类型，并且能作为泛型方法得到的实际参数类型的占位符。

抽象类与接口

　一个类可以实现多个接口，但只能继承一个抽象类。二者在应用方面也有一定的区别：接口更多的是在系统架构设计方法发挥作用，主要用于定义模块之间的通信契约。而抽象类在代码实现方面发挥作用，可以实现代码的重用。
　(1)方法
　 　抽象类可以有构造方法，而接口中不能有；
　 　抽象类可以包含静态方法，而接口中不能有；
　 　抽象类可以包含非抽象的普通方法，而接口中方法必须是public abstrac（默认也是）或default的（Java 8新特性，default public void method(){}, 可以称为Defender方法，或者虚拟扩展方法（Virtual extension methods）,实现类不必再重写该方法）；
　 　抽象类中抽象方法的访问类型可以是public、protected、和默认类型，而接口中只能是public。
　(2)成员变量
　　抽象类可以含有普通成员变量（也可为静态成员变量)，访问类型可以任意，而接口中成员变量只是是public static final,默认也是。
